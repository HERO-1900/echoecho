> 产品:双向情侣备忘录——"她说→他办→有着落",三态协议 + AI 僚机 + 语音优先。

## 1. 背景与目标

调研确认的核心故障点是:亲密关系中"说了→被忘了→吵架"这条链路,失败成本明确(单次冲突的情绪代价远超订阅费)、责任方与购买方清晰(发起方感知痛点并发起安装,执行方为"不再挨骂"付费),且高频可验证(MARKET.md 第 2 节)。24 个 Use Case 覆盖日常琐事/纪念日/大宗决策/异地时差/多元关系/硬件六类场景,反推出的 Top10 功能中,前 3 位(语音自动建卡、三态状态流、AI 僚机出方案)v1.0 已上线;**尚未落地的 P0 是第 4、5 位:场景感知的提醒时机,与反监控信息隔离**——两份 persona 调研独立收敛出同一结论:执行方最高频的卸载理由不是功能少,而是"被监控、被考核、被翻旧账"的感觉。市场侧结论同样明确:纯"记录/甜"类产品(恋爱记 3000 万注册)证明了情绪共鸣带来下载但带不来付费,EchoEcho 的付费假设必须钉死在"着落"上。

**v1.1–v1.2 的目标(按优先序)**:

1. **补齐执行方留存钩子**——办妥的正反馈回路(回响仪式)与"提醒而非骚扰"的分级机制。MARKET.md 行动建议 #5:男方 Aha 指标(7 天内办妥 2 件并收到回响)上不去,先别花一分钱营销。
2. **把"看"替代"催"做完整**——deadline 递进提醒、周期性念念,让发起方"说完就能放下"。
3. **守住三条红线**——所有新功能一票否决式过一遍设计禁区(见第 4 节)。

非目标(本期不做):硬件挂链量产(接口已预留,按 MARKET.md 走七夕众筹另立项)、原生 App 壳、模板市场、皮肤内购体系。

## 3. 新功能需求

每条含:用户故事 / 验收标准 / 依赖 / 工作量估级(S ≤ 半天,M ≤ 2 天,L > 2 天)。候选池已对照 Top10 表增删:候选池中的功能全部保留并定级;从 Top10 表补入「硬 deadline 识别与递进提醒」(#7)与「消息队列化+AI 去重分级」(#9);硬件免手持入口(#10)不在本期范围。

### 3.1 P0(v1.1 必做)

#### F-01 办妥后的"回响仪式"

- **用户故事**:作为发起方(赵晓萱,异地),TA 办妥后我收到的不是一个冷冰冰的勾选变灰,而是一条温柔的通知("你两周前提的床头灯,TA 办妥了")+ 卡片翻面动效;作为执行方(大鹏,ADHD 倾向),我办妥时看到一句明确的"你做到了",这是我婚姻里稀缺了十年的正反馈。(支撑:UC-02/04/13、Top10 #8、MARKET 男方 Aha 时刻)
- **验收标准**:
  1. 执行方点"办妥"时,本端出现完成动效 + 一句随机温暖文案(文案池 ≥ 8 条,不重复到麻木);
  2. 发起方端收到通知,文案模板为"回响"语气(含念念原文摘要 + TA 附的交待),禁止"任务已完成"式系统腔;
  3. 通知与卡片状态一致,无延迟感知(轮询周期内到达);
  4. 全链路无任何"用时 X 天""效率"类字眼。
- **依赖**:现有通知通道即可,无 schema 变更。
- **工作量**:S

#### F-02 免打扰与提醒分级(反监控底座)

- **用户故事**:作为执行方(周则明,住院医),我能设置勿扰时段,系统在我手术/开会时绝不推送,出 flow 后再合并提醒;作为发起方,我看到的永远只有任务状态,看不到 TA 已读多久没动。(支撑:UC-03/17/20 🛡️、Top10 #4/#5、执行方共性 #4)
- **验收标准**:
  1. 执行方可设每日勿扰时段(如 09:00–18:00),时段内新念念静默入队,不推通知;
  2. 勿扰结束后合并推送一条摘要("你有 3 条新念念"),而非逐条轰炸;
  3. 念念分"轻/常规/急"三级(创建时可选,默认常规;后续 F-05 由 AI 自动预判),轻念念永不主动推送,仅列表可见;
  4. 发起方端任何界面不出现:对方已读状态、已读时长、在线状态、"未响应 X 小时";
  5. 红队自查项:用发起方账号穷举所有页面与通知,确认无一处可推断对方实时行为。
- **依赖**:messages 表加 `priority` 列、用户设置加勿扰字段(仅加列);通知发送逻辑改造。
- **工作量**:M

#### F-03 Web Push 强提醒

- **用户故事**:作为执行方(陈屿),我锁屏也能收到真正的提醒,而不是要开着浏览器等 12s 轮询;这是 F-02 分级提醒和 F-06 递进提醒的运力基础。(支撑:README 路线图、UC-01/05)
- **验收标准**:
  1. iOS(PWA 加主屏)与 Android/桌面 Chrome 均可收到锁屏推送;
  2. 订阅失败时优雅降级回现有浏览器通知 + 轮询,不白屏不报错;
  3. 推送内容遵守 F-02 分级规则(勿扰时段不推、轻念念不推);
  4. 点击推送直达对应念念卡片。
- **依赖**:VAPID 密钥(Pages secret)、push 订阅表(新增表或加列)、Service Worker 改造。
- **工作量**:M

### 3.2 P1(v1.2 主体)

#### F-04 硬 deadline 识别与多级递进提醒

- **用户故事**:作为发起方(周诗涵),我说"提前一个月问你爸妈忌口",系统识别出硬节点并提前 3 周/2 周/1 周递进提醒 TA,重要的事不是提醒一次就算完;临期未动作时升级一档(急)。(支撑:UC-05/07/10/11、Top10 #7)
- **验收标准**:
  1. 建卡时 AI 从语音/文字中抽取 deadline(显式"周三前"与隐式"打烊前"均覆盖测试集 ≥ 20 条、准确率人工验收通过),执行方可一键修正;
  2. 有 deadline 的念念自动生成递进提醒计划(至少 2 档),计划对双方可见;
  3. 临期(如剩 1/4 时间)仍未"在办"时升级提醒等级,但**仅提醒执行方本人,不向发起方发任何"TA 还没办"的告警**;
  4. 过期不产生红字、不改变卡片颜色为警示色,仅执行方端温和标注"这条过期了,要不要补救?"并可一键让 AI 出补救方案。
- **依赖**:messages 表加 `deadline`、`remind_plan` 列;F-03 推送;AI runner 增加抽取 prompt。
- **工作量**:L

#### F-05 念念分类标签(AI 自动打标)

- **用户故事**:作为执行方(阿兵,月度窗口期),我一眼分清哪条是急事、哪条是心愿、哪条是随口一说;作为发起方,轻的事(朋友圈点个赞)也有类型归属而不显得小题大做。(支撑:UC-14/16、执行方共性 #2)
- **验收标准**:
  1. 标签集:急事 / 日常 / 心愿 / 纪念日 / 轻念念(可扩展),AI 建卡时自动预判,双方均可改;
  2. 列表可按标签筛选;
  3. 标签联动 F-02 提醒等级(轻念念不推送、急事可穿透常规勿扰但不穿透手术级硬勿扰)。
- **依赖**:messages 表加 `tag` 列(仅加列);AI runner 打标 prompt。
- **工作量**:S(不含联动)/ M(含 F-02 联动)

#### F-06 纪念日档案与周期性念念

- **用户故事**:作为发起方(林小满),纪念日、生日、姨妈周期这类事记一次,系统每年/每月自动生成周期念念并提前提醒 TA;"黄鹤楼旁那家日料"这种具体坐标沉淀进档案不丢失。(支撑:UC-01/06/07、MARKET 付费点 #3 的数据地基)
- **验收标准**:
  1. 支持创建周期性念念(年/月/自定义周期),到期自动生成任务卡并走 F-04 递进提醒;
  2. 纪念日档案页:小家共同的纪念日列表 + 每条关联的历史念念(去年送了什么、去了哪家店);
  3. 档案默认只展示"办成的",办砸/过期的事不进档案(证据本红线)。
- **依赖**:新增 `anniversaries` 表(或 messages 加 `recur_rule` 列);F-04。
- **工作量**:M

#### F-07 AI 僚机方案页内一键"回响给 TA"

- **用户故事**:作为执行方(阿泽),AI 给我 3 个礼物候选,我选定后一键把"我选了这个,周五到货"回传到小家消息流,不用截图另发;方案页署名永远是"AI 僚机为你准备",不是"她给你布置的作业"。(支撑:UC-08/12/13 🛡️)
- **验收标准**:
  1. 方案页内每个候选项/整页均有"回响给 TA"按钮,点击后以执行方身份发一条带方案链接的念念到消息流;
  2. 回传消息自动关联原念念卡片(发起方点开能看到"这是对哪句话的回响");
  3. 方案页署名文案检查:全站无"她要求/她布置"式措辞。
- **依赖**:现有消息 API,无 schema 变更(关联字段若无则加 `reply_to` 列)。
- **工作量**:S

#### F-08 历史念念搜索

- **用户故事**:作为执行方(徐子墨),我想起"她好像提过想看海",搜"海"就能找到那条原话和当时的上下文。(支撑:UC-06/24)
- **验收标准**:
  1. 消息流顶部搜索框,关键词命中念念正文(含语音转写文本);
  2. 结果按时间倒序,点击定位到卡片;
  3. 首版可纯前端过滤已加载消息,数据量大后再升级为 D1 LIKE/FTS 查询(留接口)。
- **依赖**:无 schema 变更。
- **工作量**:S

### 3.3 P2(v1.2 之后,验证后再排)

#### F-09 办妥率周报(只给自己看,反 KPI)

- **用户故事**:作为执行方(大鹏),每周一收到一份**只有我自己可见**的小结:"上周你接住了 5 件事",给我攒成就感;它永远不会被对方看到、被截图比较。(支撑:Top10 #8 延伸、UC-19 设计红线)
- **验收标准**:1) 周报仅执行方本人可见,产品内无任何入口供对方查看;2) 文案只做表扬与温和建议,禁用"办妥率 X%""排名""比上周下降"式语言;3) 办砸/过期的事不出现在周报正文(至多归为"要不要补救"的私密提示)。
- **依赖**:埋点数据(第 5 节);定时任务。
- **工作量**:M

#### F-10 多小家切换(家人版)

- **用户故事**:作为独生女(林小满,26 岁),我同时在"情侣小家"和"爸妈小家"里,后者用非情侣调性皮肤——粉色爱心 UI 会直接劝退我用它管理和父母的关系。(支撑:UC-18/19、执行方 persona 7 明确诉求)
- **验收标准**:1) 一个账号可加入/创建多个小家,顶部可切换;2) 小家可选调性主题(情侣/家人/通用),文案随主题切换(如"办妥回执"大字版);3) 各小家数据严格隔离。
- **依赖**:现有一对多结对模型扩展;主题系统。
- **工作量**:L

#### F-11 语音条波形与倍速

- **用户故事**:作为执行方(阿兵,窗口期 10 分钟),长语音能 2 倍速听、拖动波形跳过寒暄直达重点。
- **验收标准**:1) 语音条显示波形、支持 1x/1.5x/2x;2) 可拖动进度;3) 转写文本与语音并列展示(已有转写,补 UI 联动)。
- **依赖**:纯前端(Web Audio API),无 schema 变更。
- **工作量**:S
- **注**:候选池内唯一纯体验型条目,若 v1.1 排期有余可提前,作"今晚级"小task合适。

#### F-12 消息队列化 + AI 去重分级(离线/高噪场景)

- **用户故事**:作为执行方(阿兵,封闭训练一个月),上线看到的不是 500 条记录,而是 AI 去重合并后的"3 件事:①请假(急)②她病了想听你说话(情绪)③生日要花(有 deadline)"。(支撑:UC-16/20/22、Top10 #9)
- **验收标准**:1) 执行方离线超阈值(如 72h)后,再上线首屏展示 AI 摘要清单而非原始流;2) 摘要区分事务型/情绪型;3) 每条摘要可展开溯源到原始念念。
- **依赖**:F-05 标签;AI runner 摘要能力;离线时长判定(仅本人端使用,不外显给对方——不违反反监控:对方永远看不到)。
- **工作量**:L

#### 落选待议(记录在案,不排期)

心愿暗格/惊喜模式(UC-24)、游戏化积分兑换(UC-19 亲子专属)、可转发办妥回执大字版(UC-18,并入 F-10)、硬件挂链(另立项走众筹)。

## 4. 设计禁区(全局约束,一票否决)

来源:发起方 3 条情绪红线 + 执行方 3 条设计禁区,收敛为三总纲。**任何功能(含未来迭代)上线前逐条自查,命中即打回:**

1. **不做监工**——永不实现、永不展示:已读状态/已读时长、在线状态、地理位置、"未响应 X 小时"。进度是"看"不是"催";提醒的升级只面向执行方本人。
2. **不做证据本**——历史记录无逾期红字;办砸/过期的事默认不进任何汇总、档案、周报正文;不提供"导出对方失败记录"类能力。分手/退出小家时的数据处理遵守单方可导出自己那份的伦理设计(MARKET 付费点 #10)。
3. **不做 KPI**——全站文案禁用"办妥率排名/完成率/效率/成绩单";进度%的语言包装是经验条/着落感;不提供可炫耀式截图排行(亲子场景零容忍);周报只给本人。
4. **姿态守则**——AI 方案署名永远是"AI 僚机为你准备"而非转述发起方指令;对执行方的产品叙事是"僚机"不是"监工";对男性用户体验保持轻量聊天气泡感,避免正式待办工具的压迫感(徐子墨、UC-09)。

## 5. 非功能需求

### 5.1 隐私红线(硬约束)

- **永不采集、永不存储、永不展示**:已读时长、在线状态、地理位置。这三项在数据库 schema 层面就不设字段,杜绝"先存着以后用"。
- F-12 的离线时长判定仅在执行方本人端用于触发摘要,不落库为可查询的行为记录。
- 排班/日历类敏感信息(周则明顾虑)如未来接入,须本地/最小化处理并明示用途。

### 5.2 性能

- 语音转写:提交后 ≤ 10s 出文本(现有 Whisper 链路实测为准,超时给"稍后可看"降级)。
- 推送时延:Web Push 触发到送达 ≤ 30s;轮询兜底周期维持 12s。
- 首屏:PWA 二次打开 ≤ 2s 可交互(Service Worker 缓存)。

### 5.3 安全与工程

- 延续现状:成员 Bearer token 访问、音频/方案页按小家隔离、密钥全走 Pages secret 不进代码。
- Schema 变更纪律:优先"仅加列";任何加表/迁移属于红线操作,须用户拍板(全局宪法)。
- AI 内容免责声明保留;AI 抽取的 deadline/标签必须可被人一键修正(AI 是预填不是裁决)。

## 6. 数据埋点最小集

原则:埋点服务于 MARKET.md 的验证目标(男方 Aha、付费转化、少吵架证据),**全部为聚合内部指标,任何一项不得以任何形式展示给用户的伴侣**。

| # | 埋点 | 口径 | 服务的决策 |
|---|---|---|---|
| 1 | 双端激活漏斗 | 建小家 → 发出邀请 → 对方加入 → 双方各发 1 条 | 冷启动不对称问题(MARKET 风险 6.3) |
| 2 | 男方(执行方)Aha | 注册 7 天内:办妥 ≥ 2 件 且 收到 ≥ 1 次回响仪式 | MARKET 建议 #5,营销开闸前置条件 |
